# 方案选型
# 模块概述
1. 功能覆盖: 清晰描述模块的实现功能
2. 需求对照: 注明 a)满足需求的部分 b)暂时不实现的需求
3. 主体结构: a) 进程结构 b) 线程结构 c) 定时回调 d) 注入钩子 e) 内核驱动 f)进程通讯模型
4. 性能参数: 最大并发 最大流量 最大用户 等 (需要与需求对应)
5. 资源管理: 清晰说明各种资源的一般占用 和 最大限制 (内存 / CPU / fd / disk / 线程数 / 网络带宽 等)
6. 算法场景: 设计中, 提及到的对性能有要求的算法, 需要给出算法最差和一般情况的复杂度分析

# 结构设计 (静态结构)
1. 模块结构: 用结构图表(或者类图)描述该模块由哪些软件单元组成.
2. 模块功能: 模块结构中的每个软件单元, 说明其实现的功能

# 数据结构
1. 数据结构: 模块的所有全局数据结构, 对象成员的定义与描述
2. 配置文件: 明确定义配置文件的格式 (xml/ini/json), 每个字段的属性范围以及功能描述
3. 数据库表: 明确定义数据库使用类型 (mysql, redis), 每个表每一列的属性/范围以及功能描述
4. 文件路径: 配置文件, 数据库文件在系统中的存放路径

# 函数列表
1. 流程覆盖: 覆盖流程设计中的所有流程, 即流程中所有的步骤都有 函数对应
2. 接口描述: 结构设计中每个软件单元提供的 public 接口 和 protected 接口 (用以继承关系)
3. 函数描述: 函数的功能, { 参数和返回值 } 的 { 类型, 作用, 取值范围 }

# 详细设计
1. 逻辑实现: 以流程图, 伪代码的方式描述函数的具体流程 (不是每个一个流程图, 多个函数实现功能构成流程图)
2. 约束条件: 流程需要包含前置条件 / 后置条件, 不满足条件一般用 assert 处理
3. 异常处理: 包括 参数 / 外部读入 / 调用函数返回值 / 全局对象 的异常需要检查和处理 </br>
            处理方式包括 = 退化逻辑, 重试, 退出, 销毁对象, 忽略, 记录日志等

# 总体分析
1. 影响分析: 对系统其它模块产生的影响, 功能 / 性能 / 实现方式
2. 风险控制: 对已知技术风险有技术预研报告或风险避免说明。</br>
            需列明各个方案中存在的风险点，对每个风险点，必须有技术预研结果或风险规避说明。
3. 长期阻塞: 精灵进程 内核驱动需要给出可能导致 长期阻塞的故障场景, 并给出合理的解决方法。</br>
            可能的故障原因有 程序反复崩溃(拉不起来了), 死锁, 死循环, io高
